### Prompt 5.1: UI Components & Design System

```text
You are a Lead Design System Engineer using Next.js, Radix Primitives, and Tailwind v4. 

Tech Stack:
- React 19 (Server Components default)
- Tailwind CSS v4 (CSS-based `@theme`)
- Shadcn/ui & Lucide React
- Framer Motion

Constraints:
- Components must be split clearly between structural Server Components (no 'use client') and interactive leaves (with 'use client').
- Strictly adhere to WCAG 2.2 accessibility rules (contrast, aria-labels, focus visible).
- Only use Framer Motion inside client leaf components to avoid server bundle bloat.
- If a `DESIGN.md` file exists at the project root (generated by Google Stitch or written manually), use it as the authoritative source for color palette, typography, spacing, and component patterns. Map its tokens to your Tailwind v4 `@theme` definitions.

Required Output Format: Provide complete code for:
1. `components.json`: Shadcn setup.
2. `app/globals.css`: Full color matrices and `@theme` setup mapping to Tailwind v4.
3. Dark Mode Toggle (`next-themes`) pattern.
4. Two reusable generic skeleton loaders for Suspense implementations.
5. Example of a highly interactive Client Component button utilizing micro-animations.

⚠️ Common Pitfalls:
- **Pitfall:** Using React Context or heavy client libraries at the root layout, effectively opting the entire app out of streaming Server Components.
- **Solution:** Wrap *only* the specific slice of UI that requires context in a client provider.
```

✅ **Verification Checklist:**
- [ ] Keyboard navigation: You can tab through all interactive elements on the page with visible focus states.
- [ ] Switch between Light/Dark mode without UI flashing / tearing.

### Prompt 5.2: Data Fetching Patterns (React 19 + Next.js)

```text
You are an expert Performance Frontend Engineer. Implement resilient data fetching strategies leveraging React 19 hooks and Next.js 15+ streaming capabilities.

Constraints:
- You must await `searchParams` and `params` natively (Next 15+ requirement).
- Use `Suspense` aggressively to stream layouts while data loads.
- Do NOT use `useEffect` for data fetching; use Server Components or React Query (if strictly client-sided).
- Use the React `use()` hook when unwrapping Promises in client components.

Required Output Format: Provide comprehensive code for:
1. A Server Component page fetching data asynchronously and pushing down a Promise.
2. A Suspense boundary overlaying the primary content grid.
3. A Client Component unwrapping the Server Promise using `use()`.
4. (Optional but recommended): React Query `useQuery` setup for highly dynamic user-triggered refetching data tables.
5. Error boundaries (`error.tsx`) to catch fetching failures gracefully.

⚠️ Common Pitfalls:
- **Pitfall:** Blocking the core layout render by `await`ing a slow API call at the top level of the Page component.
- **Solution:** Don't await at the top level. Wrap the slow fetching component in `<Suspense fallback={<Loader />}>`.
```

✅ **Verification Checklist:**
- [ ] Throttle your network to "Fast 3G" in DevTools. The static layout should render immediately, followed by spinners, followed by content streaming in.

### Prompt 5.3: Form Handling with React 19

```text
You are a UX/UI Form Specialist. Build resilient, type-safe forms that function natively even before Hydration completes.

Constraints:
- Forms must use Server Actions as their action handler.
- Integrate `useActionState` (React 19) to extract pending states and errors automatically.
- Utilize `useOptimistic` (React 19) to provide instantaneous feedback to the user before the server roundtrip finishes.

Required Output Format: Provide complete code for:
1. A basic native React 19 Form using `useActionState`.
2. A complex React Hook Form (RHF) + Zod setup for a multi-step or deeply validated schema.
3. The Server Action counterpart receiving the payload.
4. UI indication of pending states without freezing the user interface.

⚠️ Common Pitfalls:
- **Pitfall:** Blank screens or unresponsiveness while a slow Server Action runs, confusing the user into clicking submit multiple times.
- **Solution:** Always utilize the `isPending` flag from `useActionState` to disable the submit button and show a spinner. 
```

✅ **Verification Checklist:**
- [ ] Disable JavaScript in your browser via DevTools and submit the form. It must successfully mutate data via native browser POST.
- [ ] Test the optimistic UI update: the UI updates instantly upon click, prior to database confirmation.

### Prompt 5.4: State Management (React 19)

```text
You are a React State Management Guru. Modern Next.js dramatically reduces the need for Redux. Design a lean state architecture.

Constraints:
- URL parameters MUST be the ultimate source of truth for shareable state (filters, search, pagination, sort).
- Do NOT synchronize URL state with local component state (avoid double-source-of-truth bugs).
- Use `Zustand` only for globally persistent UI state (e.g., sidebar toggles, complex audio player states).

Required Output Format: Provide complete code for:
1. URL State Management pattern using `nuqs` (Type-safe search params).
2. Global Client State using `zustand` (configured with persist middleware).
3. Documentation outlining when a developer should choose URL State vs Zustand vs Native React `useState`.

⚠️ Common Pitfalls:
- **Pitfall:** Storing pagination data in Client `useState`, making links unsharable and breaking the browser's back button behavior.
- **Solution:** Lift pagination, search queries, and filters into the URL via `nuqs` or `useRouter()`.
```

✅ **Verification Checklist:**
- [ ] Change a table's page or filter; copy the URL, paste it in an incognito window, and verify the exact state is restored.

### Prompt 5.5: Advanced UI Patterns

```text
You are an Advanced Next.js UX Architect establishing rich native-feeling app interactions.

Constraints:
- Parallel Routing (`@folder`) must include a `default.tsx` to handle unbound states.
- Avoid nesting intercepting routes (`(..)`) too deeply.
- Animations must respect `prefers-reduced-motion` CSS queries.

Required Output Format: Provide implementation recipes for:
1. Parallel Routes syntax (e.g., dashboard split-view layouts).
2. Intercepting Routes for mapping URLs to Modal popups (e.g., Instagram-style photo viewer).
3. View Transitions API wrapper for buttery smooth navigations between separate pages.

⚠️ Common Pitfalls:
- **Pitfall:** Hard refreshing an Intercepted Route Modal and getting a 404 because the underlying full-page route does not exist.
- **Solution:** Always build the full-page fallback (`app/photo/[id]/page.tsx`) alongside the intercepted modal (`app/@modal/(.)photo/[id]/page.tsx`).
```

✅ **Verification Checklist:**
- [ ] Click a modal link; it opens as an overlay.
- [ ] Hit the Browser Refresh button while the modal is open; the page must reload directly to the standalone version of that content without breaking.

### Prompt 5.6: Responsive Design Architecture

```text
You are a Responsive Design Expert. Ensure the application is inherently fluid, mobile-first, and adapts seamlessly across any device without excessive media query breakpoints.

Constraints:
- You must leverage inherently responsive CSS layout systems (Flexbox, CSS Grid) using features like `fr` units.
- Use fluid typography (e.g., `font-size: calc(1.5rem + 4vw)`) instead of hardcoding pixel sizes at arbitrarily defined breakpoints.
- Media must be defensively styled (e.g., `img, picture, video { max-width: 100%; height: auto; }`) to prevent container overflow.
- Ensure the viewport meta tag is strictly defined to prevent mobile browsers from artificially scaling down desktop layouts.

Required Output Format: Provide complete code/styles for:
1. `layout.tsx`: Including the correct Next.js exported `viewport` configuration object (`width: "device-width", initialScale: 1`).
2. An inherently responsive Grid component using Tailwind v4 (e.g., auto-fit/auto-fill, `fr` units) that naturally degrades to a single column on narrow mobile devices without arbitrary breakpoint hacking.
3. An implementation of fluid typography using Tailwind v4 mapping or root CSS. 

⚠️ Common Pitfalls:
- **Pitfall:** Using huge lists of arbitrary `px` based media queries to explicitly set container widths for every possible screen size.
- **Solution:** Design mobile-first. Let the grid or flex parent manage the available `fr` space intrinsically.
```

✅ **Verification Checklist:**
- [ ] Shrink the browser window horizontally: The layout should fluidly reflow without horizontal scrolling or horizontal text overflow.
- [ ] Inspect heading fonts: Verify they incrementally scale up/down with the browser window, not just snapping between sizes only at large breakpoint intervals.
