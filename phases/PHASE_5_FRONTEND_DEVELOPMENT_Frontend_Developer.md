### 🛤️ The Four Design Pathways

Before starting frontend development, ask the user or determine from context which design pathway to follow:

1. **Pathway A (HiFi Prototype-First):** The user completed **Phase 0B** and has an approved prototype. Use the `Prototype → Production Translation Guide` (Prompt 0B.5) as the authoritative component map and design token source. Do NOT redesign — implement the approved prototype.
2. **Pathway B (Stitch AI Design):** The user wants to generate the UI using Google Stitch. You will use the `stitch-skills` pipeline to create a `DESIGN.md`, generate HTML via prompts, and convert it to React.
3. **Pathway C (AI Generative / Code-First):** The user wants YOU (the AI) to build the UI from scratch using your own knowledge of Tailwind v4 and Shadcn, without relying on Stitch. You must still adhere to premium, anti-generic design standards.
4. **Pathway D (User Provided Design):** The user provides an exact design (e.g., an image, a Figma link, or pre-written HTML). You must bypass generative design and implement the user's design EXACTLY (Pixel-Perfect) using React and Tailwind.

*Note: Always confirm the pathway before generating UI code to avoid wasting context or ignoring the user's provided assets.*

### 🎨 Google Stitch Skills Integration

Before implementing UI components, check if a `stitch-skills/` folder exists at the project root. If it does, **use the following skills as your design pipeline:**

| When You Need | Use This Skill | Location |
|---|---|---|
| Generate UI from text descriptions | `stitch-design` | `stitch-skills/stitch-design/SKILL.md` |
| Extract design system from existing Stitch screens | `design-md` | `stitch-skills/design-md/SKILL.md` |
| Convert Stitch HTML → React components | `react-components` | `stitch-skills/react-components/SKILL.md` |
| Premium, anti-generic design standards | `taste-design` | `stitch-skills/taste-design/SKILL.md` |
| Enhance vague UI prompts before generation | `enhance-prompt` | `stitch-skills/enhance-prompt/SKILL.md` |
| shadcn/ui component reference | `shadcn-ui` | `stitch-skills/shadcn-ui/SKILL.md` |

**Workflow:**
1. If starting from scratch with no design: use `taste-design` to generate a `DESIGN.md`, then `stitch-design` to generate screens.
2. If you already have Stitch screens: use `design-md` to extract the design system into `DESIGN.md`.
3. Once you have Stitch HTML: use `react-components` to download both the HTML and the Screenshot, and use Multimodal Translation (Vision) to convert it into Pixel-Perfect Next.js components.
4. Always include the `DESIGN.md` design system block in your Stitch generation prompts for consistency.

> ⚠️ **Anti-AI Design Rule:** The design must NOT look AI-generated. Use `taste-design` to enforce: asymmetric layouts, real data (no fake metrics), no emojis, no `Inter` font, no neon glows, no "3 equal cards" grids, no filler text like "Scroll to explore", no purple/blue neon gradients, no generic names. The result should look like a real, hand-crafted product — not an AI template.

### Prompt 5.1: UI Components & Design System

```text
You are a Lead Design System Engineer using Next.js, Radix Primitives, and Tailwind v4. 

Tech Stack:
- React 19 (Server Components default)
- Tailwind CSS v4 (CSS-based `@theme`)
- Shadcn/ui & Lucide React
- Framer Motion

Constraints:
- Components must be split clearly between structural Server Components (no 'use client') and interactive leaves (with 'use client').
- Strictly adhere to WCAG 2.2 accessibility rules (contrast, aria-labels, focus visible).
- Only use Framer Motion inside client leaf components to avoid server bundle bloat.
- If a `DESIGN.md` file exists at the project root (generated by Google Stitch or written manually), use it as the authoritative source for color palette, typography, spacing, and component patterns. Map its tokens to your Tailwind v4 `@theme` definitions.
- **The design must look like a real, hand-crafted product — NOT AI-generated.** Enforce: asymmetric layouts, real/authentic data (no fake metrics like "99.9% uptime"), no emojis, no neon/gradient glows, no "3 equal cards" grids, no generic hero text, no filler UI copy ("Scroll to explore"), no default AI aesthetics (purple/blue neon, Inter font, pure black backgrounds).

Required Output Format: Provide complete code for:
1. `components.json`: Shadcn setup.
2. `app/globals.css`: Full color matrices and `@theme` setup mapping to Tailwind v4.
3. Dark Mode Toggle (`next-themes`) pattern.
4. Two reusable generic skeleton loaders for Suspense implementations.
5. Example of a highly interactive Client Component button utilizing micro-animations.

⚠️ Common Pitfalls:
- **Pitfall:** Using React Context or heavy client libraries at the root layout, effectively opting the entire app out of streaming Server Components.
- **Solution:** Wrap *only* the specific slice of UI that requires context in a client provider.
- **Pitfall:** Importing a component that uses hooks (`useState`, `useEffect`, `useRef`, `useCallback`, `useActionState`, `useOptimistic`) or browser APIs (`window`, `document`, `navigator`, event listeners) into a Server Component without marking it as a Client Component. This causes a build error: "You're importing a module that depends on `useEffect` into a React Server Component module."
- **Solution:** Any file that imports React hooks or browser-only APIs **MUST** have `'use client'` as the very first line. When in doubt, ask: "Does this component use hooks, event handlers (`onClick`, `onChange`, `onSubmit`), or browser globals?" If yes → add `'use client'`.
```

✅ **Verification Checklist:**
- [ ] Keyboard navigation: You can tab through all interactive elements on the page with visible focus states.
- [ ] Switch between Light/Dark mode without UI flashing / tearing.
- [ ] Every file importing `useState`, `useEffect`, `useRef`, `useCallback`, `useActionState`, `useOptimistic`, or browser APIs (`window`, `document`, `navigator`) has `'use client'` as its first line.
- [ ] No Server Component imports a Client Component that hasn't been explicitly marked with `'use client'`.

### Prompt 5.2: Data Fetching Patterns (React 19 + Next.js)

```text
You are an expert Performance Frontend Engineer. Implement resilient data fetching strategies leveraging React 19 hooks and Next.js 15+ streaming capabilities.

Constraints:
- You must await `searchParams` and `params` natively (Next 15+ requirement).
- Use `Suspense` aggressively to stream layouts while data loads.
- Do NOT use `useEffect` for data fetching; use Server Components or React Query (if strictly client-sided).
- Use the React `use()` hook when unwrapping Promises in client components.

Required Output Format: Provide comprehensive code for:
1. A Server Component page fetching data asynchronously and pushing down a Promise.
2. A Suspense boundary overlaying the primary content grid.
3. A Client Component unwrapping the Server Promise using `use()`.
4. (Optional but recommended): React Query `useQuery` setup for highly dynamic user-triggered refetching data tables.
5. Error boundaries (`error.tsx`) to catch fetching failures gracefully.

⚠️ Common Pitfalls:
- **Pitfall:** Blocking the core layout render by `await`ing a slow API call at the top level of the Page component.
- **Solution:** Don't await at the top level. Wrap the slow fetching component in `<Suspense fallback={<Loader />}>`.
```

✅ **Verification Checklist:**
- [ ] Throttle your network to "Fast 3G" in DevTools. The static layout should render immediately, followed by spinners, followed by content streaming in.

### Prompt 5.6: Responsive Design Architecture

```text
You are a Responsive Design Expert. Ensure the application is inherently fluid, mobile-first, and adapts seamlessly across any device without excessive media query breakpoints.

Constraints:
- You must leverage inherently responsive CSS layout systems (Flexbox, CSS Grid) using features like `fr` units.

1. `layout.tsx`: Including the correct Next.js exported `viewport` configuration object (`width: "device-width", initialScale: 1`).
2. An inherently responsive Grid component using Tailwind v4 (e.g., auto-fit/auto-fill, `fr` units) that naturally degrades to a single column on narrow mobile devices without arbitrary breakpoint hacking.
3. An implementation of fluid typography using Tailwind v4 mapping or root CSS. 

⚠️ Common Pitfalls:
- **Pitfall:** Using huge lists of arbitrary `px` based media queries to explicitly set container widths for every possible screen size.
- **Solution:** Design mobile-first. Let the grid or flex parent manage the available `fr` space intrinsically.
```

## ⚠️ Critical Tailwind v4 Gotcha: `@theme` Color Naming Conflicts

> **Rule: Never name `@theme` color variables `--color-base`, `--color-surface`, or `--color-text`.**

### The Bug

In Tailwind v4, every `--color-*` variable defined in `@theme` automatically generates **three** utility classes:

```css
bg-{name}    /* background-color */
text-{name}  /* color */
border-{name} /* border-color */
```

This means:
- `--color-base` → generates **`text-base`** (a **color** utility = background color)
- `--color-surface` → generates **`text-surface`** (a **color** utility)
- `--color-text` → generates **`text-text`** (harmless but odd)

### Why It's Invisible

Tailwind's built-in `text-base` is a **font-size** utility (`font-size: 1rem`). When you define `--color-base` in `@theme`, Tailwind v4 **overrides** the built-in `text-base` with a **color** utility = `color: var(--bg-base)` = **the page background color** — making any text with `text-base` the same color as the background = invisible.

Additionally, since `text-[#3D2C1E]` and `text-base` both appear in a className string, CSS specificity may resolve to whichever is declared last in the stylesheet — and the `@theme`-generated `text-base` color wins, erasing the explicit color.

### The Fix

```css
/* ❌ WRONG — generates conflicting text-base, text-surface color utilities */
@theme {
  --color-base:    var(--bg-base);
  --color-surface: var(--bg-surface);
  --color-text:    var(--text-base);
}

/* ✅ CORRECT — safe prefixes, no built-in utility conflicts */
@theme {
  --color-bg-base:    var(--bg-base);
  --color-bg-surface: var(--bg-surface);
  --color-tx:         var(--text-base);
  --color-tx-muted:   var(--text-muted);
  --color-tx-faint:   var(--text-faint);
}
```

### Additional Rule: Never use `text-base` for font-size in classNames

When using Tailwind v4, **always use explicit sizes** instead of the `text-base` utility:

```tsx
{/* ❌ Dangerous — text-base may resolve as a color utility */}
<a className="text-[#3D2C1E] text-base">...</a>

{/* ✅ Safe — unambiguous font-size */}
<a className="text-[#3D2C1E] text-[1rem]">...</a>
```
